# E2Eテスト（Playwright）

「全ての機能が正しく動作するか」を確認するためのテストフロー。
今すぐ自動化できる範囲はPlaywrightで自動テスト化し、リアルタイム通信や
メール認証など自動化が難しい範囲は下記チェックリストで手動確認する。
チェックリストは将来的な自動化のロードマップ（todo）としても使う。

## 0. 事前準備

1. バックエンド一式を起動（SPACE-server側）
   ```
   docker compose up
   ```
2. フロントの開発サーバーを起動
   ```
   npm run dev
   ```
3. E2E専用のテストアカウントを用意する（**本番ユーザーを流用しない**）
   - 一般ユーザー1名（メール認証済み・利用規約同意済みの状態にしておく）
   - 管理者1名
4. `e2e/.env.test.example` を `e2e/.env.test.local` にコピーして値を埋める
5. 環境変数を読み込んでテスト実行
   ```
   set -a; source e2e/.env.test.local; set +a
   npm run test:e2e        # ヘッドレス実行
   npm run test:e2e:ui     # UIモード（おすすめ・1件ずつ確認しやすい）
   npm run test:e2e:report # 直前の実行のHTMLレポートを開く
   ```

`setup` プロジェクトがGraphQL APIに直接ログインしてトークンを取得し、
`user` / `admin` プロジェクトの `storageState` を生成する（ブラウザ操作での
ログインではないため高速・安定）。実際のログインUI自体は `anon` プロジェクトで
別途検証する。

## 1. 自動テストの構成

```
e2e/
  auth.setup.ts          ログインしてstorageStateを保存（setup project）
  support/                共通ヘルパー（env, API直叩き, 利用規約モーダル回避）
  anon/                   未ログイン状態のテスト（ログイン画面・遷移）
  user/                   ログイン済みユーザーのテスト
  admin/                  ログイン済み管理者のテスト
```

| ファイル | 内容 |
|---|---|
| `anon/user-login.spec.ts` | ユーザーログイン画面の表示・エラー・成功時遷移、新規登録/パスワード再設定/お問い合わせへの遷移 |
| `anon/admin-login.spec.ts` | 管理者ログイン画面の表示・エラー・成功時遷移 |
| `user/navigation-smoke.spec.ts` | 動的IDが不要なユーザー向け画面が認証済みで表示できるか一通り確認 |
| `user/post-flow.spec.ts` | 投稿作成→詳細表示→返信→いいね→削除までの一連の操作 |
| `user/community-flow.spec.ts` | コミュニティ作成→一覧表示→チャットルーム表示 |
| `user/search.spec.ts` | ユーザー検索（該当なしケース） |
| `admin/navigation-smoke.spec.ts` | 動的IDが不要な管理画面が認証済みで表示できるか一通り確認 |

⚠️ **既知の制約**
- `community-flow.spec.ts` はコミュニティを作成するが、削除APIが存在しないため
  実行するたびにテストデータが残る。頻繁に実行する場合は管理者画面の
  コミュニティ管理から手動で整理するか、サーバー側に削除APIを追加することを検討する。
- 利用規約の新バージョンが公開され未同意の場合、保護ページ全面に同意モーダルが
  被る。`support/terms.ts` の `dismissTermsConsentModalIfPresent` で自動同意を
  試みるが、ドキュメント取得が失敗する環境では効かないことがある。
- DM・コミュニティチャットはSSE/WebSocketのリアルタイム通信に依存しており、
  CI環境でのタイミング起因のflakinessを避けるため、メッセージ送受信そのものは
  まだ自動化していない（画面が開けることだけ確認）。

## 3. 自動化ロードマップ（次にやると良いこと）

1. **フォーム要素に `data-testid` を付与**：現状はテキスト/placeholderベースで
   セレクタを組んでいるため、文言変更に弱い。特に編集系フォーム（プロフィール、
   コミュニティ編集、管理者編集系）を自動化する際は先にtestidを振ると安定する。
2. **詳細系（`:id` を含むルート）の自動化**：API直叩きでテスト用データ（投稿・
   コミュニティ・通知・お知らせ）を作成するヘルパーを `support/` に増やし、
   作成したIDで `/posts/:id` 等に直接遷移するテストを追加する。
3. **リアルタイム機能（DM・コミュニティチャット）**：2クライアント
   （2つの `browserContext`）でメッセージ送受信を検証するテストを追加する。
   SSE/WebSocketの接続待ちは `expect.poll` 等でタイムアウトを明示的に短くする。
4. **メール系フロー（登録OTP・パスワード再設定）**：開発環境のSMTPをMailHog等の
   テスト用メールキャッチャーに向け、APIでOTPを取得するヘルパーを用意する。
5. **CI連携**：GitHub Actionsで `docker compose up -d` →ヘルスチェック→
   `npm run test:e2e` を実行するワークフローを追加する（`reporter: 'github'` は
   `playwright.config.ts` で既にCI時のみ有効化済み）。
6. **コミュニティ削除のクリーンアップ**：テストで作成したコミュニティを
   後始末できるAPI（またはテスト専用のリセットスクリプト）をサーバー側に用意する。
